Si vous avez encore des doutes quant à la pertinence de l’utilisation des tests dans la construction d’application complexes nous allons rectifier tout cela ici.
Plutôt que de vous parler directement des raisons pour lesquelles utiliser les tests est vital, on va parler des quelques situations où on peut s’en passer.

- POC (Proof Of Concept)

    Ce n’est pas nécessaire de faire les tests immédiatement.
    A vous de juger si vous en avez besoin pour la partie développement.
    -> En revanche si le POC est validé il est conseillé d’ajouter des tests. 

- Code à jeter

     Logique similaire au POC, les tests ne sont pas nécessaires si le code ne va pas être mis à l’épreuve du temps. 
     En revanche écrire des tests peut aider à développer, vous serez juge. 

- Contraintes de temps (à prendre avec des pincettes)



Alors…, voilà le sujet sensible.
Dans l’imaginaire collectif, le fait d’écrire des tests fait « perdre du temps ». 
Mais tout est relatif à l’ensemble de référence n’est ce pas ?

Il convient donc de se poser la question: Perdre du temps à quelle échéance ?


- Sur les heures de développement ?
    Pendant la phase de développement, il y a certes plus de code à écrire.
     A ne pas oublier qu’écrire des tests aide à écrire du meilleur code.
     Il est facile de casser ce qu’on est entrain de construire; avoir une couverture de test, même partielle, aide à éviter les erreurs et les oublis.

- Sur les heures de vie du code que vous avez écris ?
     On peut être tout de suite sur un ensemble de référence bien plus large.
     En effet à chaque fois que votre nouveau code sera exécuté, il sera mis à l’épreuve des bugs.

 - Sur sa future modification et la suite de sa vie ?
    Ne soyons pas naïf, le propre d’une architecture logicielle est très souvent d’évoluer pour rester en phase avec son objectif métier qui lui, change.
    Vos tests aideront le prochain développeur qui aura la charge de modifier, de refactorer, d’adapter, de supprimer votre code pour le bien-être de l’application.
